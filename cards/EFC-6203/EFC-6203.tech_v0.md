# Plano Técnico — EFC-6203

> Gerado em: 2026-04-28

## Contexto

O refinamento define que `ItinerarioFormativo` deve referenciar diretamente `Rede`, `AnoLetivo` e `Agrupamento`, eliminando a entidade intermediária `ItinerarioFormativoRedeSerie`. A implementação abrange schema de banco, entidades, mappings EF Core, repositório, serviços e testes. O frontend não precisa ser alterado pois o contrato de API (DTOs) já reflete a estrutura desejada.

## Convenções aplicáveis

1. Sem EF Migrations — toda alteração de schema é feita via script SQL manual em `c:/projects/scripts-db-pedagogico/`.
2. DTOs referenciam entidades por `Hash` (Guid), nunca por `Id` numérico.
3. Queries de leitura usam `QueryableAtivos()` para filtrar soft-deleted.
4. Serviços usam primary constructor (C# 12) para injeção de dependência.
5. Repositórios implementam interfaces granulares (`IRepositoryGet`, `IRepositoryInsert`, etc.).
6. Identificadores públicos expõem `Hash`; `Id` numérico fica com `[JsonIgnore]`.

## Restrições e riscos

- **Breaking change de schema**: a remoção de `ItinerarioFormativoRedeSerie` exige que o script SQL seja executado antes do deploy do código. Rollback deve estar planejado (backup da tabela antes do drop).
- **Views e relatórios SQL**: buscar em `scripts-db-pedagogico/` por referências a `ItinerarioFormativoRedeSerie` antes do deploy — views ou relatórios que usem essa tabela falharão.
- **CA-02 — proteção de Rede/AnoLetivo/Agrupamento na edição**: definir se esses campos são imutáveis após criação ou se há regra de negócio para alteração. O serviço atual não trata esse cenário explicitamente.
- **Índices**: após adicionar FKs diretas em `ItinerarioFormativo`, criar índice composto em `(RedeId, AnoLetivoId, AgrupamentoId)` para performance das queries de filtro.
