## R — Requirements

### Problema

`ItinerarioFormativo` associa-se a `Rede`, `AnoLetivo` e `Agrupamento` de forma indireta, passando por `ItinerarioFormativoRedeSerie` — uma entidade intermediária que não agrega valor, pois um itinerário sempre pertence a exatamente uma combinação desses três valores. A mesma indireção existe em `ItinerarioFormativoRedeSerieDisciplina`, que referencia o itinerário via `ItinerarioFormativoRedeSerie` em vez de referenciar `ItinerarioFormativo` diretamente. O objetivo é remover essa camada desnecessária, tornando o modelo direto, as queries mais simples e as regras de negócio menos sujeitas a erros.

### Definição de Pronto

- [ ] Script SQL de migração executado e validado em staging antes do deploy de código
- [ ] Rollback script preparado (backup de `ItinerarioFormativoRedeSerie` antes do DROP)
- [ ] Entidade `ItinerarioFormativoRedeSerie` removida do domain e do banco
- [ ] `ItinerarioFormativo` possui FK direta para `Rede`, `AnoLetivo` e `Agrupamento`
- [ ] `ItinerarioFormativoRedeSerieDisciplina` possui FK direta para `ItinerarioFormativo`
- [ ] Todos os serviços operam sem referência a `ItinerarioFormativoRedeSerie`
- [ ] CA-01 a CA-08 cobertos por testes de integração passando
- [ ] Zero referências a `ItinerarioFormativoRedeSerie` remanescentes no código C#
- [ ] Views e stored procedures em `scripts-db-pedagogico` verificadas e atualizadas se necessário
- [ ] Módulos externos (`notas`, `frequencia`, `aulas`, `barramento-pedagogico`) verificados — sem dependência direta na tabela removida

### Critérios de Aceite

| # | Critério |
|---|---------|
| CA-01 | Criar itinerário → `RedeId`, `AnoLetivoId`, `AgrupamentoId` gravados direto em `ItinerarioFormativo` |
| CA-02 | Editar itinerário → `Rede`, `AnoLetivo`, `Agrupamento` preservados corretamente |
| CA-03 | Listagem filtrada por `(Rede, AnoLetivo, Agrupamento)` retorna somente itinerários da combinação |
| CA-04 | Disciplinas vinculadas ao itinerário possuem `ItinerarioFormativoId` direto (sem intermediária) |
| CA-05 | Inativação funciona normalmente (soft delete via `Disable()`) |
| CA-06 | Sistema impede nome duplicado na mesma `(Rede, AnoLetivo, Agrupamento)`; permite em combinações diferentes |
| CA-07 | Períodos disponíveis filtrados por `(Rede, AnoLetivo, Agrupamento)` do itinerário |
| CA-08 | Itinerários disponíveis para alunos filtrados por `Rede` |

### Fora de Escopo

- Contrato de API (DTOs já expõem `HashRede`, `HashAnoLetivo`, `HashAgrupamento` — não mudam)
- Alterações funcionais no frontend (fluxo de tela permanece igual)
- Modificações em módulos externos — apenas verificação de ausência de dependência

## A — Approach

### Estratégia

Executar a simplificação em 6 fases sequenciais com dependência rígida: o script SQL (Fase 1) deve ser aplicado em staging **antes** do deploy de qualquer código C#, pois o código novo depende das colunas novas e não compilará corretamente contra o schema antigo. As fases 2–5 seguem a ordem natural de dependência de camadas (Domain → Infra → Services). A Fase 6 (testes) cobre todos os CAs ao final. O contrato de API não muda — os DTOs já usam `Hash` como identificador externo — então o frontend e os consumers externos não precisam de alterações funcionais.

### Decisões de Design

| Decisão | Escolha | Alternativa descartada | Motivo |
|---------|---------|----------------------|--------|
| Manter nome da entidade `ItinerarioFormativoRedeSerieDisciplina` | Manter | Renomear para `ItinerarioFormativoDisciplina` | Evitar renomeação de tabela no banco em escopo desta task; pode ser feito em task separada |
| Validação de unicidade de nome | `private` no service | FluentValidation | Convenção do projeto: sem FluentValidation novo (ver Norms) |
| Índice composto `(Rede, AnoLetivo, Agrupamento)` | Criar no script SQL | Deixar sem índice | Performance das queries de filtro — CA-03, CA-07, CA-08 dependem desse filtro |
| Colunas de FK no banco | `HasColumnName("Rede")` sem sufixo `Id` | `HasColumnName("RedeId")` | Convenção do projeto (ver Norms) |

### Operação 1 — Script SQL de migração de schema

**Status:** ✅ concluída — 2026-04-29

**Repositório:** `scripts-db-pedagogico`  
**Arquivo:** `Processos manuais/EFC-6203-simplificar-itinerario-formativo.sql`

Criar script único com as seguintes etapas **em ordem**:

```sql
-- 1. Adicionar colunas com DEFAULT temporário para permitir UPDATE
ALTER TABLE ItinerarioFormativo
    ADD Rede        INT NOT NULL DEFAULT 0,
        AnoLetivo   INT NOT NULL DEFAULT 0,
        Agrupamento INT NOT NULL DEFAULT 0;

-- 2. Popular colunas via join com a estrutura intermediária
UPDATE IF
SET    IF.Rede        = RS.Rede,
       IF.AnoLetivo   = RS.AnoLetivo,
       IF.Agrupamento = RS.Agrupamento
FROM   ItinerarioFormativo IF
JOIN   ItinerarioFormativoRedeSerie IFRS ON IFRS.ItinerarioFormativo = IF.Id
JOIN   RedeSerie RS ON RS.Id = IFRS.RedeSerie;

-- 3. Remover DEFAULT temporários e adicionar FOREIGN KEY constraints
ALTER TABLE ItinerarioFormativo DROP CONSTRAINT DF_ItinerarioFormativo_Rede;        -- nome real pode variar
ALTER TABLE ItinerarioFormativo DROP CONSTRAINT DF_ItinerarioFormativo_AnoLetivo;
ALTER TABLE ItinerarioFormativo DROP CONSTRAINT DF_ItinerarioFormativo_Agrupamento;
ALTER TABLE ItinerarioFormativo
    ADD CONSTRAINT FK_ItinerarioFormativo_Rede        FOREIGN KEY (Rede)        REFERENCES Rede(Id),
        CONSTRAINT FK_ItinerarioFormativo_AnoLetivo   FOREIGN KEY (AnoLetivo)   REFERENCES AnoLetivo(Id),
        CONSTRAINT FK_ItinerarioFormativo_Agrupamento FOREIGN KEY (Agrupamento) REFERENCES Agrupamento(Id);

-- 4. Índice composto para queries de filtro (CA-03, CA-07, CA-08)
CREATE INDEX IX_ItinerarioFormativo_Rede_AnoLetivo_Agrupamento
    ON ItinerarioFormativo (Rede, AnoLetivo, Agrupamento);

-- 5. Adicionar FK direta em ItinerarioFormativoRedeSerieDisciplina
ALTER TABLE ItinerarioFormativoRedeSerieDisciplina
    ADD ItinerarioFormativo INT NOT NULL DEFAULT 0;

-- 6. Popular nova coluna
UPDATE IFRSD
SET    IFRSD.ItinerarioFormativo = IFRS.ItinerarioFormativo
FROM   ItinerarioFormativoRedeSerieDisciplina IFRSD
JOIN   ItinerarioFormativoRedeSerie IFRS ON IFRS.Id = IFRSD.ItinerarioFormativoRedeSerie;

-- 7. Remover constraint de FK antiga e DEFAULT temporário
ALTER TABLE ItinerarioFormativoRedeSerieDisciplina DROP CONSTRAINT DF__IFRSD__ItinerarioFormativo;
ALTER TABLE ItinerarioFormativoRedeSerieDisciplina DROP CONSTRAINT FK_IFRSD_ItinerarioFormativoRedeSerie;  -- nome real pode variar
ALTER TABLE ItinerarioFormativoRedeSerieDisciplina
    ADD CONSTRAINT FK_IFRSD_ItinerarioFormativo FOREIGN KEY (ItinerarioFormativo) REFERENCES ItinerarioFormativo(Id);

-- 8. Backup e DROP de ItinerarioFormativoRedeSerie
SELECT * INTO ItinerarioFormativoRedeSerie_BACKUP_EFC6203 FROM ItinerarioFormativoRedeSerie;
DROP TABLE ItinerarioFormativoRedeSerie;
```

**Critério de conclusão:** script executa sem erros em staging; `SELECT COUNT(*) FROM ItinerarioFormativo WHERE Rede = 0` retorna 0; tabela `ItinerarioFormativoRedeSerie` não existe mais; tabela backup criada.

## N — Norms
> Convenções do repositório `estrutura-pedagogica`. Toda implementação deve seguir estas regras.

### Convenções específicas deste card (do `tech.md`)

- **Sem EF Migrations** — alterações de schema via script SQL manual em `scripts-db-pedagogico`
- **FK column naming** — `HasColumnName("Rede")`, não `HasColumnName("RedeId")` — sem sufixo
- **DTOs referenciam por `Hash` (Guid)** — nunca por `Id` inteiro
- **`QueryableAtivos()`** — nunca filtrar `Ativo` manualmente com `.Where(x => x.Ativo)`
- **Primary constructor (C# 12)** — sem `this._field = field` em serviços e repositórios
- **Interfaces granulares** — `IRepositoryInsert`, `IRepositoryGet`, `IRepositoryDisable`, etc.
- **Projeção obrigatória via `.Select()`** — nunca retornar entidade inteira em queries de leitura
- **Sem FluentValidation novo** — validações de negócio como `private` no service
- **Async obrigatório** — todo acesso a banco retorna `Task<T>`

### Convenções gerais do repositório

- Propriedades de entidade: sempre `virtual` (EF Core proxy)
- `[ExcludeFromCodeCoverage]` em entidades e controllers
- XML doc (`/// <summary>`) em métodos públicos de controllers
- Retorno de serviços: `ServiceResult` ou `ServiceResult<T>` — nunca lançar exceção para erros de negócio
- Autorização: `[RequiredAuthorization(Funcionalidade.Configurador_Itinerario_Formativo)]` — sem alteração esperada
- Namespace: `EstruturaPedagogica.[Camada].[Feature]`
- Testes: `BaseTest` + `AddGlobalization()` + `AddInMemoryDbContext()`; `UsuarioAutenticadoRepository` sempre mockado com `_usuarioId = 99999`
