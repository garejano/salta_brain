# Plano Técnico — EFC-6203

> Gerado em: 2026-04-28

## Contexto

O refinamento define que `ItinerarioFormativo` deve referenciar diretamente `Rede`, `AnoLetivo` e `Agrupamento`, eliminando a entidade intermediária `ItinerarioFormativoRedeSerie`. O impacto atravessa todas as camadas: schema de banco, entidades, interfaces, mappings EF Core, repositórios, serviços e testes. O contrato de API (DTOs já expõem `HashRede`, `HashAnoLetivo`, `HashAgrupamento`) não muda; o frontend provavelmente não precisa de alterações funcionais.

## Convenções aplicáveis

1. **Sem EF Migrations** — alterações de schema são scripts SQL manuais em `c:/projects/scripts-db-pedagogico/`.
2. **FK column naming** — a coluna recebe o nome da entidade relacionada sem sufixo `Id` (ex.: `HasColumnName("Rede")`, não `HasColumnName("RedeId")`).
3. **DTOs referenciam por `Hash` (Guid)** — nunca por `Id` numérico.
4. **`QueryableAtivos()`** — nunca filtrar `Ativo` manualmente.
5. **Primary constructor (C# 12)** — sem `this._field = field` em serviços e repositórios.
6. **Repositórios com interfaces granulares** — `IRepositoryInsert`, `IRepositoryGet`, `IRepositoryDisable`, etc.
7. **Projeção obrigatória via `.Select()`** — nunca retornar a entidade inteira em queries de leitura.
8. **Sem FluentValidation novo** — validações em método `private` no serviço.
9. **Executável assíncrono obrigatório** — todo acesso a banco retorna `Task<T>`.
10. **Autorização** — `[RequiredAuthorization(Funcionalidade.Configurador_Itinerario_Formativo)]` nos controllers (sem alteração esperada).

## Critérios de Aceite

- **CA-01:** Criar Itinerário Formativo associando diretamente Rede, AnoLetivo e Agrupamento.
- **CA-02:** Editar Itinerário Formativo preserva Rede, AnoLetivo e Agrupamento.
- **CA-03:** Listagem filtrada por Rede + AnoLetivo + Agrupamento retorna somente os itinerários da combinação.
- **CA-04:** Disciplinas do itinerário continuam vinculadas diretamente ao itinerário (sem entidade intermediária).
- **CA-05:** Inativação continua funcionando (soft delete via `Disable()`).
- **CA-06:** Impede criação de dois itinerários com o mesmo nome na mesma combinação (Rede, AnoLetivo, Agrupamento).
- **CA-07:** Seleção de períodos continua filtrada por Rede, AnoLetivo e Agrupamento.
- **CA-08:** Seleção de itinerários para alunos continua filtrada por Rede.

### Fase 1 — Script SQL de migração de schema _(infraestrutura)_

**Status:** pendente

Criar um único script SQL em `c:/projects/scripts-db-pedagogico/` com as seguintes etapas (em ordem):

1. `ALTER TABLE ItinerarioFormativo ADD Rede INT NOT NULL DEFAULT 0, AnoLetivo INT NOT NULL DEFAULT 0, Agrupamento INT NOT NULL DEFAULT 0` — adicionar colunas com defaults temporários para permitir o `UPDATE`.
2. `UPDATE` populando `Rede`, `AnoLetivo`, `Agrupamento` via join com `ItinerarioFormativoRedeSerie` → `RedeSerie`.
3. Remover os `DEFAULT` temporários e adicionar `FOREIGN KEY` constraints para as três colunas.
4. Criar índice composto em `(Rede, AnoLetivo, Agrupamento)` na tabela `ItinerarioFormativo` para performance das queries de filtro.
5. `ALTER TABLE ItinerarioFormativoRedeSerieDisciplina ADD ItinerarioFormativo INT NOT NULL DEFAULT 0`.
6. `UPDATE ItinerarioFormativoRedeSerieDisciplina` populando `ItinerarioFormativo` via join com `ItinerarioFormativoRedeSerie`.
7. Remover constraint de FK para `ItinerarioFormativoRedeSerie` da tabela `ItinerarioFormativoRedeSerieDisciplina`.
8. `DROP TABLE ItinerarioFormativoRedeSerie` (após remover todas as constraints que a referenciam).

> Script deve ser validado em staging antes de qualquer deploy de código. Criar rollback script com backup da tabela antes do drop.

## Restrições e riscos

- **Breaking change de schema**: A remoção de `ItinerarioFormativoRedeSerie` exige que o script SQL seja executado e validado em staging **antes** do deploy do código novo. Preparar rollback com backup da tabela antes do `DROP`.
- **Views e relatórios SQL**: Buscar em `scripts-db-pedagogico/` e no banco por referências a `ItinerarioFormativoRedeSerie` — views ou stored procedures que usem essa tabela falharão após o drop.
- **Módulos consumidores externos** (`notas`, `frequencia`, `aulas`, `barramento-pedagogico`): Verificar se algum desses repositórios consulta `ItinerarioFormativoRedeSerie` diretamente antes do merge.
- **CA-02 — imutabilidade de Rede/AnoLetivo/Agrupamento**: O comportamento atual precisa ser verificado — se esses campos são imutáveis após criação, o serviço deve rejeitar tentativas de alteração; se mutáveis, a lógica de `Update()` precisa garantir consistência.
- **Índice composto**: Após a data migration, criar índice em `(Rede, AnoLetivo, Agrupamento)` para não degradar performance das queries de filtro.
- **Dados em produção**: Todos os `ItinerarioFormativo` existentes têm exatamente uma `ItinerarioFormativoRedeSerie`, então o `UPDATE` de data migration não deve gerar nulos. Validar com `SELECT COUNT(*) FROM ItinerarioFormativo WHERE Rede IS NULL` após o script.
